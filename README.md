# cursor-full-spec

This repository is a **code generation experiment**: the application code here was auto-generated by Cursor using skills and a strictly ordered sequence of prompts. This README describes that experiment, its artefacts, and objectives—not the generated application itself.

## What this experiment is

- **Code generation experiment** — The code in this repo was produced by Cursor (LLM-assisted) from a high-level specification, without hand-written application logic.
- **Skills-driven** — Generation was guided by Cursor Skills that encode reusable engineering practices (see below).
- **Prompt-driven** — A fixed sequence of prompts was followed in strict order to establish patterns early and keep each step scoped.
- **Result** — The experiment has been very successful: the generated codebase is of good quality and **required zero human editing**.
- **Status** — Outcomes are very encouraging as experimental results; this is not presented as proven production code.

## Providing and using skills

Cursor Skills were written and registered to steer generic, reusable engineering practices. Examples include:

- Dependency injection via interfaces at system boundaries
- Deterministic fake test-double implementations for those interfaces in tests
- Code structure that supports separation-of-concerns and readability
- How to refactor
- How to apply intelligent renaming of functions, types, files, etc.
- Test isolation
- Test rigour
- Test uniformity
- Test writing (patterns)
- Linting requirements for acceptance

The set of skills used in this experiment can be seen at:  
https://github.com/peterhoward42/cursorlib.git

## Strictly ordered prompts

- A **strictly ordered sequence of prompts** was used to:
  - Encourage key shapes and patterns to take root and form precedent early.
  - Package the work into relatively small steps so the model could get each domain right before moving on.
- Prompts were written in advance and are included in this repo as examples and as a record (e.g. under `docs/prompts/`).
- Each prompt was run in a **new, dedicated chat context** to conserve LLM memory and avoid precision drift from long conversations.

## High-level complete system spec overview

A conceptual entry point and fallback reference was provided via a **high-level overview of the complete generated system behaviour**. That overview described the intended API, storage, and analysis behaviour so the generator had a single source of truth.
